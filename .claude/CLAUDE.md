# 개발 규칙 (CLAUDE.md)

이 문서는 모든 에이전트가 반드시 준수해야 하는 개발 규칙이다.

## 설계 원칙

- **SOLID**: SRP, OCP, LSP, ISP, DIP 준수
- **KISS**: 단순하고 명확하게
- **YAGNI**: 현재 필요한 것만 구현
- **하드코딩 금지**: 매직 넘버, 매직 스트링 사용 금지
- **Bottom-Up**: 항상 작은 규모(단위 함수, 타입)부터 설계하고 점차 큰 규모(모듈, 도메인)로 조합
- **중복 금지**: 동일 로직이 여러 곳에 작성되지 않도록 항상 유의하고, 공통 로직은 즉시 추출
- **유지 보수**: 유지 보수가 손쉬운 구조가 곧 좋은 코드

## 확장성 원칙

- 모든 수치 데이터(서열별 이성 상실, 소화 주기, 기능 등급 가산치 등)는 설정/데이터 파일로 외부화
- 새로운 경로, 구역, 이벤트, 아이템은 데이터 추가만으로 확장 가능하도록 설계
- 플러그인/레지스트리 패턴으로 새 시스템을 등록할 수 있는 구조 유지
- 하드코딩된 분기(if/switch)가 아닌 데이터 기반 분기 선호

## 코드 품질

- **명료한 네이밍**: 변수, 메서드, 클래스명은 역할과 의도가 즉시 드러나도록 작성 (약어 지양, 도메인 용어 사용)
- **최소 단위 메서드**: 하나의 메서드는 하나의 책임만 가지며, 20줄 내외의 간결한 코드를 권장. 초과 시 추출을 고려
- 의존성 최소화
- 코드 품질 준수 (lint, format)

## 테스트

- TDD 설계: 테스트를 먼저 작성
- given/when/then 패턴 사용
- 각 도메인은 독립적으로 테스트 가능해야 함
- E2E 테스트를 필수적으로 사용

## 도메인 분리

- 각 도메인은 `src/domains/<도메인명>/` 디렉터리로 분리
- 각 디렉터리에 `README.md`로 구조와 목적을 명시
- 도메인 간 직접 의존 금지
- 병렬 에이전트 작업 가능 구조 유지

## 문서화

- 요청 및 수정사항을 `/logs/` 디렉터리에 기록
- 로그 파일명 규칙: `yyyy-mm-dd-index-contents.md` (예: `2026-03-06-01-integration-design.md`)
  - `index`: 같은 날짜 내 순번 (01, 02, ...)
  - `contents`: 작업 내용을 영문 kebab-case로 요약
- 같은 요청에 대한 내용은 하나의 로그 파일에 기록
- 해당 도메인의 `README.md` 준수

## 설계 토론 (페르소나 리뷰)

설계 및 구현 계획 단계에서 최종 결론은 다음 페르소나들의 토론을 거쳐 도출한다.

### 페르소나 구성

| 페르소나 | 역할 | 관점 |
|----------|------|------|
| **Architect** | 구조 설계자 | 도메인 경계, 확장성, 의존 방향, 계층 분리 |
| **Player** | 최종 사용자 대변 | 게임플레이 경험, 직관성, 재미, 밸런스 |
| **Skeptic** | 비판적 검토자 | YAGNI/KISS 위반, 과잉 설계, 숨은 복잡도, 성능 리스크 |
| **Maintainer** | 유지보수 담당자 | 테스트 용이성, 가독성, 변경 비용, 디버깅 난이도 |

### 토론 규칙

1. **증거 기반**: 모든 주장은 구체적인 코드(파일 경로, 함수명, 라인)를 증거로 제시
2. **구조화된 발언**: 각 페르소나는 `[주장] → [근거: 코드 증거] → [제안]` 형식으로 발언
3. **합의 도출**: 전원 동의 또는 다수결로 결론. 반대 의견이 있으면 트레이드오프를 명시
4. **기록 의무**: 토론 내용 전체를 `/logs/` 디렉터리에 해당 일자 문서로 기록

### 적용 시점

- 새로운 도메인 또는 시스템 추가 시
- 기존 아키텍처에 영향을 주는 변경 시
- 구현 방식이 2가지 이상 존재하여 판단이 필요할 때
- 단순 버그 수정이나 데이터 추가 등 명확한 작업에는 생략 가능

## 구현 후 리뷰

코드 구현 완료 후, 커밋 전에 다음 항목을 비판적으로 검토한다:

- **도메인 경계**: 도메인 간 직접 의존이 생기지 않았는가
- **타입 일관성**: 새 타입이 기존 패턴(readonly, 불변)과 일관되는가
- **하위 호환**: 기존 호출부가 깨지지 않는가
- **데이터 기반 설계**: 하드코딩된 분기가 추가되지 않았는가
- **테스트 커버리지**: 새 로직에 대한 테스트가 충분한가
- **불필요한 복잡도**: KISS/YAGNI 위반이 없는가

문제 발견 시 커밋 전에 수정한다.

## 커밋 컨벤션

- 1줄: 주요 변경사항 요약
- n줄: 변경사항 내역
- **매 요청 완료 시 반드시 커밋**: 구현 → 테스트 → 리뷰 → 커밋 순서로 마무리
- 변경사항을 쌓아두지 않고, 요청 단위로 즉시 커밋
